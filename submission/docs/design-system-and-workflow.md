## 1. エグゼクティブサマリー

### 何を作ったか
「本質のClaude Code 完全攻略 7dayブートキャンプ」の販売スライドを、ターゲット別に3パターン（A：副業・案件獲得訴求／B：法人・業務改善訴求／C：初心者安心訴求）で自動生成するシステムを、Python + python-pptx + Claude API で構築しました。同じ入力から何度でも再現性をもって生成できる状態にしています。

### 主要な判断
**1. 公式システムを使わず、ゼロから組みました。**
ブートキャンプ特典の cc-slides・pptx-agent は認識していましたが、お題は「販売スライドを自動生成するシステムを作る」であり、公式と同じレイヤーの新しいシステムを組むことが求められていると解釈したためです。ただし「少ない工数でセットアップできる」という公式の思想は踏襲しています。

**2. 3パターン固定に絞り、自由テイスト変更は次フェーズに送りました。**
「シンプルで」「モダンで」と自然言語で指示して任意のテイストで生成する機能も4時間ほど試しましたが、座標ベースの設計では再現性が保てないと判断しました。削った時間は3パターンの一発生成の品質向上に振り替えています。

**3. AIモデルの使い分けは、走りながら気づいた学びでした。**
最初はClaude CodeのSonnet 4.6で全部やろうとしていましたが、途中でクオリティの限界にぶつかり、月曜以降は「設計＝Opus 4.7／実装＝Sonnet／壁打ち＝ChatGPT」という役割分担に切り替えています。詳細は4章・7章、最大の反省点は10章に記載しています。

## 2. デザイン方針・コンセプト

### 「読む資料」ではなく「見せる資料」だと気づいたところから始まりました

最初にデザイナーエージェントを使って自動生成したスライドを見て、「なんか違う」と感じました。その違和感をChatGPTで壁打ちして言語化したところ、根本的な問いに辿り着きました。

**「これは読む資料か、見せる・感じさせる資料か？」**

販売スライドは、登壇者が話しながら見せるものです。文字で説明するのではなく、1枚で1つの感情を引き出すのが目的です。この認識から、構成・デザインを全面的に見直しました。

### 1スライド1メッセージ・1画面1感情

採用したのは2つの原則です：

- **1スライド1メッセージ：** 1枚に伝えることは1つだけ。複数の主張を詰め込まない
- **1画面1感情：** 「怖さ」「期待」「安心」「決断」など、1枚ずつ感情を積み上げる

数字を主役にする、余白を意図的に残す、登壇前提で読まなくても伝わる構成にする——これらは全て「見せる資料」という認識から派生したルールです。

### 感情導線

3パターン共通で、以下の流れを骨格にしています：

```
問題提起（現状への違和感）
  ↓
危機感（このままだとどうなるか）
  ↓
解決策の提示（Claude Codeという選択肢）
  ↓
信頼・権威（講師紹介）
  ↓
再現性（実績・受講生の声）
  ↓
具体的な学習内容（アクションマップ）
  ↓
価値の積み上げ（特典・パッケージ）
  ↓
価格訴求（払えそう感・ROI）
  ↓
CTA（今すぐ申し込む理由）
```

パターンによって言葉や強調点は変わりますが、感情導線の骨格は共通にしています。

### 採用後の運用イメージから逆算した設計

「販売スライドを作る」だけでなく、「採用された後にどう使われるか」までを想定した上で設計しました。具体的な仮説と判断は8章「こだわった3点」に記載します。

### 公式システムの思想（導入容易性）の踏襲

ブートキャンプで配布されている公式システム（cc-slides、pptx-agent）の存在は認識していました。今回はそのまま使わない判断をしました（理由は6章に記載）が、公式の「少ない工数でセットアップできる」設計思想は意識的に踏襲しました。

機能はゼロから組みましたが、思想は公式をリスペクトしています。

## 3. スライドの構成・流れの設計意図

### 3パターンの感情軸の違い

同じサービスでも、誰に見せるかによって「刺さる言葉」「感じてほしい感情」「デザインのトーン」は変わります。3パターンで狙った感情はこのように整理しました：

| パターン | ターゲット | 狙った感情 | 主軸 |
|---|---|---|---|
| A：副業・案件獲得訴求型 | AI副業に興味がある個人 | 今動けば先行者になれる | 月10万円案件 / 7日 / 0.1% |
| B：法人・業務改善訴求型 | 中小企業経営者・管理職 | 自社の業務に使えそう | 業務効率化 / 自動化 / 内製化 |
| C：初心者安心訴求型 | AI・PC初心者 | 自分でもできそう | 未経験OK / 日本語OK / 質問し放題 |

### ターゲット設計の経緯

正直に書くと、開発開始時点で「ターゲットを誰に絞るか」という視点は持っていませんでした。仕様書のサービス情報を読んだ後、ChatGPT・Claudeに壁打ちして「販売スライドにはターゲット別の訴求軸があるべき」という気づきを得ました。そこからAIにターゲット候補を出してもらい、提示された3つ（副業／法人／初心者）をそのまま採用しました。

ただし「AIが出した3つを、何も考えず採用した」わけではありません。採用した瞬間に自分の中で腑に落ちた感覚がありました：

- **副業層：** 自分自身が副業目的でスキルプラスに入った当事者なので、ペルソナを描く必要がなく、刺さる訴求が想像できました
- **初心者層：** 副業層とセットで考えた時、パソコン未経験からの参入経路として必要だと判断しました
- **法人層：** 上記2つとは違う軸として「業務改善」というアプローチを試したかった

### 一番自信があるのは副業層、課題は初心者層と法人層

3パターンの中で「これは売れる」と一番自信を持って言えるのは副業層（Pattern A）です。理由は明確で、**自分自身がペルソナだから**。時間のない副業層に「自動化スキル」を訴求する論理は、データやペルソナ分析ではなく、自分の実体験から書けました。

逆に、初心者層は「パソコン未経験」への訴求に少しハードルを感じた箇所があります。法人層は実装したものの、このサービスの実際の主購入層に法人がどれだけ含まれるかは今も疑問が残っています。ここは10章「自己評価」とも繋がる課題として認識しています。

### 表紙コピーの設計：3パターンそれぞれの一発勝負

販売スライドは1枚目で読み手の感情を掴めるかが勝負だと考えました。3パターンそれぞれ、異なる入口を設計しています。

**Pattern A：「7日で、月10万円案件を狙えるスキルへ。」**

副業層が最も気にしている「時間（7日）」と「金額（月10万円）」を表紙の一文で同時に提示しました。

ここで重要な判断をしました。最初AIが提案してきたのは「月10万円の案件を取れます」という断定型でしたが、これを「狙えるスキルへ」に書き換えました。理由は単純で、**人任せで出来るほど甘くないから**です。

販売スライドの世界では断定型コピーが刺さるとされますが、誇大広告にならない範囲で訴求するのは、佐藤さん・アドネスというブランドにも沿うと判断しました。月商規模の大きい組織は、コピーの誠実性に敏感なはずです。

**Pattern B：「AI導入に、外注は必要ない。」/「7日間で、自社のAI活用を内製化する。」**

中小企業の経営者・管理職が一番嫌がるのは「外注コストの増大」と「丸投げによる依存」だと考えました。それを表紙で先回りして打ち消し、「内製化できる」という前向きな選択肢を提示する流れにしています。

「業務改善」「効率化」という抽象的な言葉ではなく、「外注 vs 内製化」という意思決定ワードで切り込んだのは、決裁者の判断軸に合わせるためです。

**Pattern C：「プログラミングは、いりません。」/「日本語で話しかけるだけで動く。」**

このコピー1本で、見る人の心理を3段階で動かすつもりで設計しました：

1. **「プログラミングって稼げるのは知ってるけど、自分には出来ないな」** — 初心者が普段抱えている諦め
2. **「いらないの？」** — 想定外の言葉で立ち止まらせる
3. **「いらないってことは、プログラミング覚えなくてもプログラミングが出来る人並みのスキルが身につくのかも？」** — 期待を先回りで提示

サブコピーで「日本語で話しかけるだけで動く」と即座に答え合わせをすることで、3段階目の期待を裏切らずに着地させています。

これは2章の「1スライド1メッセージ・1画面1感情」を、コピー側で実装した形です。

### コピーの誠実性は3パターン共通の方針

Pattern Aで「狙える」を選んだのと同じ判断軸で、B・Cの表紙コピーも誇大表現を避けました。Bは「内製化する」と事実ベースで、Cは「日本語で話しかけるだけ」と機能ベースで止めています。詳細は8章「こだわった3点」にも記載します。

## 4. 使用したAI・ツール

### 技術スタック

| ツール / AI | 用途 |
|---|---|
| Claude Code（CLI） | システム全体の設計・コーディング・ファイル操作 |
| Claude API（claude-opus-4-7） | スライドコンテンツのJSON生成 |
| Opus 4.7（Claude.ai） | 設計判断・仕様書解析・上流の壁打ち |
| Sonnet 4.6（Claude Code内） | 実装・コード修正 |
| ChatGPT | デザイン・構成の壁打ち・改善アドバイス |
| Python + python-pptx | PowerPointファイルの自動生成 |
| PyYAML | service_info.yaml の読み込み |

### AIモデルの使い分け（走りながら確立した役割分担）

最初はClaude CodeのSonnet 4.6だけで全部やろうとしていました。チャットでお願いすればどんなシステムも作れると思っていたためです。

しかし途中でクオリティの限界にぶつかりました。3パターンのスライドは生成できるようになりましたが、テイストを変えると文字が被る・配置が崩れる・同じ指示でも結果がバラつく、といった現象が連発しました。

ChatGPTに相談したところ「設計フェーズはOpus 4.7を使うべき」というアドバイスを受け、ようやくモデルの使い分けという観点を理解しました。月曜以降は以下の役割分担で進めています：

| フェーズ | 使うAI | 理由 |
|---|---|---|
| 設計判断・仕様解析 | Opus 4.7 | 深い読み解きが必要なため |
| 実装・コード修正 | Sonnet 4.6（Claude Code） | 速度と量で勝負する作業 |
| 上流の壁打ち | ChatGPT | 別視点での切り返しが欲しいため |

ChatGPTはコーディングには使わず、「どの情報を削るか」「感情導線をどう設計するか」という上流の設計相談だけに使いました。

「AIに頼む」と「AIに何を頼むかを設計する」は別物だと、ここで初めて理解しました。Claude Codeブートキャンプが推奨する「AIにやらせる」設計の本質は後者にあると気づいたのは、月曜日の夜です。この気づきを最初から持っていたら動き方が変わっていたという反省は、10章「自己評価」に最大の反省点として記載します。

### service_info.yaml による情報の一元管理

サービス情報（料金・カリキュラム・講師情報など）は service_info.yaml に集約しました。スライドのコピーや訴求軸を変えたいときも、コードを触らずにYAMLを書き換えるだけで全パターンに反映される設計です。

これは「のぼる以外の人がこのシステムを使う場面」を想定した判断でした。コードを読める人でないと運用できない設計だと、採用された後にチームに渡せません。

### Claude APIのレスポンス安定化の工夫

Claude APIからのJSONレスポンスにマークダウン記号（` ```json … ``` `）が混入することがあったため、`[` 〜 `]` を正規表現で抽出するロジックを追加しました。これでテキスト整形のブレを吸収し、毎回同じ形式のスライドが生成される状態を確保しています。

## 5. プロンプト連鎖・エージェント構成

### 全体のデータフロー

現在のシステムは、以下の流れで動いています：

```
[入力]
service_info.yaml(サービス情報)
  +
パターン指定(A / B / C / ALL)

    ↓

[Claude API(Opus 4.7)]
各スライドのコピー・構成を JSON で生成
(1スライド1メッセージの原則に従い生成)

    ↓

[python-pptx]
JSON をもとにスライドを自動描画
色・フォント・レイアウトはパレット定義から参照

    ↓

[出力]
.pptx ファイル(パターンごとに別ファイル)
```

### パターン選択の対話フロー

`python3 generate_v6.py` を実行すると、まず以下の選択画面が表示されます：

```
どのパターンを生成しますか？

  [A] 副業・案件獲得訴求(黒×パープル)
  [B] 法人・業務改善訴求(白×青)
  [C] 初心者安心訴求(白×グリーン)
  [ALL] 全パターン一括生成
```

ユーザーがA/B/C/ALLのいずれかを入力すると、該当パターンだけが生成されます。CLIコマンド引数（例：`python3 generate_v6.py B`）でも直接指定可能です。

これは「使う人に選ばせる」ためのフローです。チームに渡したとき、用途に応じて1パターンだけ取り出せる方が運用しやすいと考えました。

### CLAUDE.md でClaude Codeにルールを与える

プロジェクトフォルダの CLAUDE.md には、Claude Codeへの指示書を記載しています。

主なルール：

- 「スライド作成して」と言われたら、A/B/C/ALL のどれを生成するかをユーザーに必ず確認する
- 自己判断でALL（全パターン）を勝手に実行しない
- ユーザーの返答を待ってからコマンドを実行する

これは「Claude Code が独走しない」ための歯止めです。Claude Code は便利ですが、何も指示しないと「全部やった方がいいだろう」と判断して走り出すことがあります。CLAUDE.md でルールを明示しておくことで、運用の予測可能性が上がります。

### エージェント構成は単純（現状の限界）

正直に書くと、現在のシステムは「単一プロンプト → JSON生成 → 描画」というシンプルな構成です。仕様書にある「プロンプト連鎖・エージェント構成」を本格的に組み込むのは、Phase2の構想として残しています（詳細は11章「Phase2構想」）。

具体的には：

- 「販売スライド構成を考えるマーケチーム」エージェント
- 「スライド作成デザインチーム」エージェント

これらを連鎖させる多エージェント設計が、次に取り組みたい課題です。

## 6. 設計判断：公式システムを使わずゼロから組んだ理由

### 認識していた公式システム

ブートキャンプの特典として配布されている公式システムを認識していました。

- **cc-slides：** Gemini API による画像生成スライド
- **pptx-agent：** Google スプレッドシート駆動の PPTX 自動生成

どちらも「自分が作りたいスライドを作るための汎用インフラ」として、すぐに使えるものです。

### お題の解釈：別レイヤーのシステムが求められている

これらを使わずにゼロから組むことにしたのは、お題の読み解きの違いによります。

仕様書のお題は「販売スライドを自動生成するシステムを作る」。

この「システムを作る」という言葉を、私は次のように解釈しました。

**「公式システムと同じレイヤーの新しいインフラを、販売スライドという用途に特化して作ることが求められている」**

公式システムは「作る人が自分のスライドを作るための汎用インフラ」であって、それをそのまま使って提出するのは、お題の表面しか拾っていない応募になると考えました。仕様書6.4「当たり前の正解ではなく、応募者ならではの工夫があるか」に対する、自分なりの回答として、独自に組む選択をしました。

### 独自に組んだことで得られたもの

ゼロから設計したことで、以下を実現できました：

- **販売心理に基づく構成：** 感情導線を全スライドに通すことができた
- **ターゲット別の感情導線設計：** 副業・法人・初心者で訴求軸を変えられた
- **1スライド1メッセージの徹底：** ルールをコード側で強制できた
- **言語化の余地：** 設計判断のすべてを自分の判断軸で説明できる

これらは、公式システムをカスタマイズする形では到達しにくい設計密度だと考えています。

### 思想は公式をリスペクトしています

機能はゼロから組みましたが、公式システムが持つ「少ない工数でセットアップできる」という思想は意識的に踏襲しました。

具体的には：

- service_info.yaml で情報を一元管理し、コードを触らずに内容を変更可能にした
- CLAUDE.md にルールを明示し、Claude Code に渡せばすぐに動く状態にした
- 対話フロー（A/B/C/ALL選択）で、誰でも使える操作性にした

公式システムを使わなかった判断は、公式を否定するものではありません。お題の解釈として別レイヤーが必要だったという判断であり、その上で公式の思想は今回のシステムにも引き継いでいます。

## 7. 開発中の気づきと判断の進化

### 出発点：「AIに頼めば何でも作れる」という前提

ブートキャンプ参加前から Cursor を触っていたこともあり、「Cursor や Claude Code にチャットでお願いすれば、どんなシステムも大体作れる」という前提で今回のタスクに入りました。

実際、3パターンのスライドを生成するシステム自体は、Sonnet 4.6 の Claude Code でほぼ作れました。

### 限界に直面した瞬間

詰まったのは、自由テイスト変更の実装に入ったときです。

「シンプルで」「モダンで」と自然言語で指示して任意のテイストで生成できる機能を試したところ、以下の現象が連発しました：

- 文字の被り（特定パターンで発生）
- レイアウトの崩れ
- 同じ指示でも結果がバラつく
- 味気ない出力になる

「同じ入力で何度でも再生成できること」という仕様書の必須要件に対し、Sonnet モードの自由テイスト変更では再現性が保証できないと判断しました。

約4時間試行を重ねましたが、突破口が見えませんでした。

### ChatGPT で壁打ちして気づいたこと

行き詰まったときに ChatGPT に状況を整理して相談したところ、返ってきた指摘は「設計フェーズは Opus 4.7 を使うべき」でした。

それまで「AIに頼む」しか考えていませんでしたが、**「どのAIに何を頼むか」という選別こそが重要**だと、ここで初めて気づきました。

### 判断の進化：「AIに頼む」と「AIに何を頼むかを設計する」は別物

この経験を経て、自分の中の理解が変わりました。

| Before | After |
|---|---|
| AIに頼めば何でも作れる | AIごとに得意領域がある |
| 1つのAIで全部やる | フェーズごとにAIを使い分ける |
| 「AIにやらせる」= AIに丸投げ | 「AIにやらせる」= どのAIに何を任せるかを設計する |

Claude Codeブートキャンプが推奨する「AIにやらせる」設計の本質は、後者にあると理解しました。

ここから設計判断は Opus 4.7、実装は Claude Code（Sonnet）、上流の壁打ちは ChatGPT という役割分担に切り替え、現在に至っています。

### この気づきは、評価タスク自体への学びでもありました

この学びを得たのが Day4（月曜日）。最大の反省は、Day1 の時点で仕様書の読み込みを Opus 4.7 に依頼しなかったことです。詳細は10章「自己評価」の最大の反省点として記載します。

## 8. こだわった3点

### こだわり①：採用後の運用イメージを織り込んだ設計

3パターンを設計する時、ある仮説を置きました。

「みかみさん・佐藤さんがブートキャンプで使われているスライドは、淡い紫のバーと白地のシンプル構成。これは"話して伝える"前提の設計。だとすれば、今回作るスライドは、シンプル性ではなく訴求力に振った方がいい」

そしてもう一段先のことも想定しました。もし右腕として採用された場合、「1期生がこのスライドを作ってくれました」というコンセプトで佐藤さんが実際にセミナーで使われる可能性があると思い、その時に転用できる状態にしておく必要があると考えました。

これは「販売スライドを作る」という作業ではなく、「採用後どう使われるか」まで読んだ設計判断です。

### こだわり②：コピーの誠実性

3パターンすべてで、断定型の誇大コピーを意図的に避けました。

代表例として、Pattern A の帯コピー「月10万円案件を狙えるスキルへ」。

最初 AI が提案してきたのは「月10万円の案件を取れます」という断定型でした。これを「狙えるスキルへ」に書き換えた根拠は、**人任せで出来るほど甘くない**という自分の倫理観です。

販売スライドの世界では断定型コピーが刺さるとされます。しかし誇大広告型のコピーは、月商規模の大きい組織やブランドにとっては毀損リスクの方が大きいと考えました。佐藤さん・アドネスというブランドにも沿うはずです。

Pattern B では「内製化する」と事実ベースで、Pattern C では「日本語で話しかけるだけ」と機能ベースで止めています。3パターン共通の方針です。

### こだわり③：3パターンの一発生成の品質

「同じ入力で何度でも再生成できること」という仕様書の必須要件に対し、A/B/C それぞれが安定して高品質に生成される状態を確保しました。

ピクセル単位の調整、文字の重なり、配置の崩れを潰すために、ターゲットスコープを 3 パターン固定に絞りました。自由テイスト変更を諦めた時間を、ここに集中投下しています（詳細は9章「削る判断」）。

「全部できるシステム」より「3パターンを毎回同じクオリティで出せるシステム」を選んだ判断です。

### 1つだけ残すなら、こだわり①を選びます

3つのこだわりの中で、もし1つだけ残すならこだわり①（採用後の運用イメージを織り込んだ設計）を選びます。理由は、相手の使い方を想像してから作るという視点を、自分の中で一番大事にしたいと思っているからです。

## 9. 削る判断

### 削った①：自由テイスト変更（戦略的に次フェーズへ送りました）

自由テイスト変更（自然言語で「シンプルで」「モダンで」と指示して任意のテイストで生成する機能）は、最初から構想にありました。仕様書の「スライド自動生成システム」という言葉から、単に紹介・販売スライドを作るだけでなく、テイストや指示を変えて何度でも再生成できるシステムを作るべきだと感じたためです。

しかし約4時間の試行で、座標ベースの設計では再現性が保てないと判断しました。

諦めるトリガーは3つありました：

1. ChatGPT壁打ちで「ここは捨てるべき」とアドバイスを得たこと
2. 仕様書「何を削るか」が評価軸に明記されていたことを思い出したこと
3. 試した時間は言語化ドキュメントの素材として無駄にならないと気づいたこと

この判断は「諦めた」というより「戦略的に次フェーズへ送った」という位置づけにしています。詳細は11章「Phase2構想」に記載します。

### 削った時間は、3パターンの品質向上に再投資しました

自由テイスト変更を削った瞬間、削った時間を3パターンの一発生成の品質向上に振り替えました。

結果として、A/B/Cそれぞれが安定して高品質に生成される状態を確保できました。仕様書の必須要件「同じ入力で何度でも再生成できる」を完全に満たせています。

「削った」だけで終わらせず、「削った分、ここを厚くした」と言える状態にすることを意識しました。

### 削った②：色・構成判断（AIによるプロの判断を採用しました）

色彩設計と販売スライド構成の細部については、AI提案をそのまま採用しました。

これは「自分で判断することを削った」のではなく、**「AIに蓄積されたプロデザイナー・マーケターの判断を採用した」という戦略選択**として位置づけています。

色彩心理や販売構成の定石は、何十年もかけて積み上げられた知見がAIに蓄積されています。ここに自分の素人判断を入れるより、AI提案を採用する方が品質が安定します。

これは仕様書6.1「AIに聞いて自分でやる」ではなく「AIにやらせる」設計の実践そのものです。アドネスが「みかみさん→佐藤さん→受講生」と知見をアウトソースする組織であるように、自分も色・構成の判断をAIにアウトソースしました。

### 「こだわる／削る」の判断軸

何をこだわり、何を削るかを判断するとき、自分の中で以下3つの基準を併用していました：

- **右腕として評価される観点になっているか**
- **時間内に提出できるか**
- **自分の判断軸を見せられる場所か**

これらは「AIにやらせる」という大方針の中で、自分が判断軸を持って動く領域を絞るためのフィルターでした。

## 11. Phase2構想

今回のシステムは「3パターン固定の販売スライド生成」に特化しましたが、1ヶ月の時間があれば以下の順序で拡張したいと考えています。

### 優先順位

**1位：複数AIによるエージェント構成**

仕様書にある「プロンプト連鎖・エージェント構成」への直接的な進化です。

- 「販売スライド構成を考えるマーケチーム」エージェント
- 「スライド作成デザインチーム」エージェント

これらを連鎖させる多エージェント設計を組み込みます。現状の「単一プロンプト → JSON生成 → 描画」というシンプルな流れを、専門領域ごとに分解したい狙いです。

**2位：自由テイスト変更の実装**

今回諦めた機能への再挑戦です。レイアウトエンジンの再構築が前提となるため、座標ベース設計から脱却した上で、自然言語でのテイスト指示に応えられる仕組みを目指します。

**3位：ターゲット入力対応**

現在はターゲットがA/B/Cの3パターン固定ですが、ユーザー入力でターゲットそのものを定義できる構成に拡張します。「30代女性向け」「医療業界向け」など、サービス側のニーズに合わせた柔軟な生成を可能にします。

**4位：生成後の自動微調整**

出力されたスライドを再度AIが検査し、文字被り・配置崩れを自動修正するループを実装します。これが実現すれば、自由テイスト変更も品質を保ったまま動かせる見込みです。

### 最終的に到達したいシステム像

最終ゴールは、以下のようなシステムです。

**チャット欄での質疑応答形式で、過去に開催実績のあるセミナー情報を読み込み、対話しながら販売スライドを自動生成できるシステム。**

具体的には：

- **入力：** セミナー情報（既存ドキュメント・録画文字起こし・参加者データ等）
- **対話：** 「どんなターゲット向け？」「どこを訴求したい？」を質疑応答
- **出力：** 販売スライド一式

これは Claude Code・Claude API・RAG（検索拡張生成）・複数エージェント構成の組み合わせで実現可能と考えています。

「販売スライドを作る作業」を完全に自動化する一歩手前まで持っていく構想です。
